# Net Libraries — Roadmap
#
# General-purpose networking libraries for KDOS Forth.
# Same (addr len) cursor model as json.f, markup/core.f, css.f.
# Zero-copy where possible, no hidden allocations.
#
# These libraries factor out the reusable patterns from tools.f
# (SCROLL system, ~990 lines) and bsky.f (~2000 lines) into proper,
# standalone, composable libraries.  After this refactor, tools.f
# becomes a thin consumer of net/, and bsky.f drops ~500 lines of
# hand-rolled HTTP/URL code in favour of library calls.
#
# Files:
#   utils/net/url.f       — URL parsing, encoding, query strings
#   utils/net/base64.f    — Base64 encode/decode (RFC 4648)
#   utils/net/headers.f   — HTTP header builder/parser
#   utils/net/http.f      — HTTP/1.1 client (GET/POST/PUT/DELETE)
#   utils/net/websocket.f — WebSocket client (RFC 6455)
#   utils/net/ftp.f       — FTP/FTPS client
#   utils/net/gopher.f    — Gopher client (type-0 text)
#   utils/net/tftp.f      — TFTP client (RRQ/DATA/ACK)
#   utils/net/ntp.f       — NTP time sync (RFC 5905 SNTP)
#   utils/net/irc.f       — IRC client (RFC 2812)
#   utils/net/rabbit.f    — Rabbit protocol client (burrow browser)
#   utils/net/smtp.f      — SMTP client (RFC 5321, STARTTLS)
#
# PREFIX:  NET-   (shared infra — errors, polling, buffer helpers)
#          URL-   (url.f)
#          B64-   (base64.f)
#          HDR-   (headers.f)
#          HTTP-  (http.f)
#          WS-    (websocket.f)
#          FTP-   (ftp.f)
#          GOPH-  (gopher.f)
#          TFTP-  (tftp.f)
#          NTP-   (ntp.f)
#          IRC-   (irc.f)
#          RBT-   (rabbit.f)
#          SMTP-  (smtp.f)
#
# Internal prefixes follow _PREFIX- convention throughout.
#
# =====================================================================
#  What KDOS Already Provides (Available — Not Re-Implemented)
# =====================================================================
#
# §16 Network Stack:
#   Ethernet — ETH-BUILD, ETH-SEND, ETH-RECV
#   ARP      — ARP-LOOKUP, ARP-RESOLVE, ARP-RESPONDER
#   IPv4     — IP-BUILD, IP-SEND, IP-RECV, NEXT-HOP
#   ICMP     — PING, PING-IP
#   UDP      — UDP-SEND, UDP-RECV
#   DHCP     — DHCP-START
#   DNS      — DNS-RESOLVE, DNS-LOOKUP
#   TCP      — TCP-CONNECT, TCP-LISTEN, TCP-SEND, TCP-RECV,
#              TCP-CLOSE, TCP-STATUS (4 TCB slots)
#   TLS 1.3  — TLS-CONNECT, TLS-SEND, TLS-RECV, TLS-CLOSE,
#              TLS-SNI-HOST, TLS-SNI-LEN
#              Cipher suites: AES-256-GCM+SHA3 / AES-128-GCM+SHA256
#              Post-quantum: ML-KEM-512 hybrid key exchange
#
# §17 Socket API:
#   SOCKET, BIND, LISTEN, ACCEPT, CONNECT, SEND, RECV, CLOSE
#
# Crypto:
#   AES-256/128-GCM, SHA-3, SHA-256, HMAC, HKDF, X25519,
#   ML-KEM-512, TRNG, CRC32/64
#
# Memory:
#   ALLOCATE, FREE, RESIZE (heap allocator)
#   Arena allocator (ARENA-CREATE, ARENA-ALLOT, ARENA-DESTROY)
#
# Other:
#   Filesystem (MP64FS), RTC, Timer, COMPARE, MOVE, FILL
#
# =====================================================================
#  What tools.f Currently Provides (To Be Refactored)
# =====================================================================
#
# tools.f SCROLL system currently contains ~500 lines of net code:
#   - URL parsing (proto://host[:port]/path) — ~120 lines
#   - HTTP GET request builder — ~30 lines
#   - HTTP response parsing (header-end, Content-Length) — ~80 lines
#   - HTTPS GET via TLS — ~50 lines
#   - FTP/FTPS client — ~60 lines
#   - TFTP client — ~40 lines
#   - Gopher client — ~30 lines
#   - Dispatch + public API — ~60 lines
#
# After net/ libraries exist, tools.f SCROLL becomes ~80 lines:
#   URL-PARSE → HTTP-GET / HTTP-GET-TLS → SCROLL-BUF
#   Plus FTP/TFTP/Gopher dispatched to net/ftp.f etc.
#
# =====================================================================
#  What bsky.f Currently Re-Implements (Could Use net/)
# =====================================================================
#
# bsky.f contains ~400 lines of HTTP infra that net/ replaces:
#   - HTTP POST request builder (BSK-BUILD-POST) — net/http.f
#   - HTTP GET request builder with auth (BSK-BUILD-GET) — net/http.f
#   - HTTP response parser + status code (BSK-PARSE-RESPONSE) — net/http.f
#   - Chunked transfer decoder (_BSK-DECHUNK) — net/http.f
#   - TLS send/recv loop (BSK-XRPC-SEND) — net/http.f
#   - URL encoding (URL-ENCODE) — net/url.f
#   - DNS caching + SNI setup — net/http.f
#   - HBW receive buffer management — net/http.f
#
# After net/ exists, bsky.f drops to ~1200 lines of pure
# application logic (JSON, auth, TUI).
#

#  Architecture — Dependency Graph
# =====================================================================
#
#   url.f ────────┐
#                  │
#   base64.f ──┐   │

#

#   ntp.f     — standalone (uses UDP directly)

# =====================================================================
#  §1 — url.f — URL Parsing & Encoding
# =====================================================================
#
#  PROVIDED akashic-url
#
#  Generalises tools.f's URL parser.  Adds query string parsing,
#  fragment support, and URL encoding/decoding.
#
#  ── Layer 0 — Percent Encoding ──
#
#    URL-UNRESERVED? ( c -- flag )
#        RFC 3986 §2.3: ALPHA / DIGIT / - . _ ~
#
#    URL-ENCODE      ( src len dst max -- written )
#        Percent-encode src into dst buffer.  Returns bytes written.
#        Encodes everything outside unreserved set.
#
#    URL-ENCODE-COMPONENT ( src len dst max -- written )
#        Same as URL-ENCODE but also encodes / : @ etc.
#        For query parameter values.
#
#    URL-DECODE      ( src len dst max -- written )
#        Decode percent-encoded string.  %XX → byte.
#
#    _URL-HEX-DIGIT  ( n -- char )     0-15 → '0'-'9','A'-'F'
#    _URL-HEX-VAL    ( char -- n|-1 )  hex char → 0-15
#
#
#  ── Layer 1 — URL Parsing ──
#
#    URL-PARSE       ( addr len -- ior )
#        Parse URL into component variables.  Returns 0 on success.
#        Sets: URL-SCHEME, URL-HOST, URL-PORT, URL-PATH, URL-QUERY,
#              URL-FRAGMENT, URL-USERINFO.
#
#    Component storage (cursor pairs into original URL string):
#      URL-SCHEME     ( -- addr )  VARIABLE pair: addr + len
#      URL-SCHEME-LEN ( -- addr )
#      URL-HOST       CREATE 64 ALLOT     hostname copy buffer
#      URL-HOST-LEN   VARIABLE
#      URL-PORT       VARIABLE             numeric port
#      URL-PATH       CREATE 256 ALLOT    path copy buffer
#      URL-PATH-LEN   VARIABLE
#      URL-QUERY      CREATE 256 ALLOT    query string (after ?)
#      URL-QUERY-LEN  VARIABLE
#      URL-FRAGMENT   CREATE 64 ALLOT     fragment (after #)
#      URL-FRAG-LEN   VARIABLE
#      URL-USERINFO   CREATE 64 ALLOT     user:pass@
#      URL-USER-LEN   VARIABLE
#
#    URL-DEFAULT-PORT ( -- port )
#        Default port for current scheme (80, 443, 21, 70, 69,
#        1965, 6667, 25, etc.)
#
#    Scheme constants:
#      URL-S-HTTP     = 0
#      URL-S-HTTPS    = 1
#      URL-S-FTP      = 2
#      URL-S-FTPS     = 3
#      URL-S-TFTP     = 4
#      URL-S-GOPHER   = 5
#      URL-S-RABBIT   = 6
#      URL-S-IRC      = 7
#      URL-S-IRCS     = 8
#      URL-S-SMTP     = 9
#      URL-S-NTP      = 10
#      URL-S-WS       = 11
#      URL-S-WSS      = 12
#
#
#  ── Layer 2 — Query String ──
#
#    URL-QUERY-NEXT  ( a u -- a' u' key-a key-u val-a val-u flag )
#        Iterate key=value pairs in query string.
#        Handles & and ; separators.  Percent-decodes keys and values
#        in place.  Flag = -1 found, 0 end.
#
#    URL-QUERY-FIND  ( a u key-a key-u -- val-a val-u flag )
#        Find a specific query parameter by key name.
#
#    URL-QUERY-BUILD ( dst max -- )
#        Start building a query string into dst buffer.
#        Initialiser — call before URL-QUERY-ADD.
#
#    URL-QUERY-ADD   ( key-a key-u val-a val-u -- )
#        Append "key=value&" to query builder (percent-encoded).
#
#    URL-QUERY-RESULT ( -- addr len )
#        Return the built query string (without trailing &).
#
#
#  ── Layer 3 — URL Building ──
#
#    URL-BUILD       ( dst max -- )
#        Start building a URL into dst buffer.
#
#    URL-BUILD-SCHEME ( scheme-id -- )
#        Append "http://", "https://", etc.
#
#    URL-BUILD-HOST  ( host-a host-u -- )
#    URL-BUILD-PORT  ( port -- )    omits if default for scheme
#    URL-BUILD-PATH  ( path-a path-u -- )
#    URL-BUILD-QUERY ( query-a query-u -- )
#    URL-BUILD-FRAG  ( frag-a frag-u -- )
#
#    URL-BUILD-RESULT ( -- addr len )
#
#
#  Error handling:
#    URL-ERR            VARIABLE
#    URL-FAIL           ( err-code -- )
#    URL-OK?            ( -- flag )
#    URL-CLEAR-ERR      ( -- )
#    URL-E-MALFORMED    = 1
#    URL-E-UNKNOWN-SCHEME = 2
#    URL-E-OVERFLOW     = 3
#
#
# =====================================================================
#  §2 — base64.f — Base64 Encode / Decode (RFC 4648)
# =====================================================================
#
#  PROVIDED akashic-base64
#
#  Essential for HTTP auth headers, MIME attachments, data URIs.
#  No dependencies.
#
#    B64-ENCODE      ( src slen dst dmax -- written )
#        Standard Base64 encode.  Writes into dst, returns
#        bytes written.  Pads with '=' as needed.
#
#    B64-DECODE      ( src slen dst dmax -- written )
#        Decode Base64.  Ignores whitespace.  Returns decoded
#        byte count.  Sets B64-ERR on invalid input.
#
#    B64-ENCODE-URL  ( src slen dst dmax -- written )
#        URL-safe Base64: uses - and _ instead of + and /,
#        no padding.  (RFC 4648 §5)
#
#    B64-DECODE-URL  ( src slen dst dmax -- written )
#        Decode URL-safe Base64.
#
#    B64-ENCODED-LEN ( n -- m )
#        Calculate encoded length for n input bytes.
#        m = (n + 2) / 3 * 4.
#
#    B64-DECODED-LEN ( n -- m )
#        Calculate decoded length for n encoded bytes.
#        m = n / 4 * 3 (approximate, minus padding).
#
#  Error handling:
#    B64-ERR         VARIABLE
#    B64-E-INVALID   = 1    invalid character
#    B64-E-OVERFLOW  = 2    output buffer too small
#
#  Internal:
#    _B64-CHAR       ( 6bit -- char )   index → Base64 alphabet
#    _B64-VAL        ( char -- 6bit|-1 ) char → index
#    _B64-TABLE      — 64-byte lookup (A-Z, a-z, 0-9, +, /)
#
#
# =====================================================================
#  §3 — headers.f — HTTP Header Builder & Parser
# =====================================================================
#
#  PROVIDED akashic-http-headers
#
#  Handles the boilerplate of HTTP header construction and parsing
#  that's currently duplicated between tools.f and bsky.f.
#
#  ── Building ──
#
#    HDR-RESET       ( -- )
#        Clear the header buffer for a new request.
#
#    HDR-SET-OUTPUT  ( addr max -- )
#        Set the output buffer for header building.
#        Default: an internal 4 KB buffer.
#
#    HDR-METHOD      ( method-a method-u path-a path-u -- )
#        Append "METHOD /path HTTP/1.1\r\n".
#        method-a/u is "GET", "POST", "PUT", "DELETE", "PATCH", etc.
#
#    HDR-GET         ( path-a path-u -- )    shortcut: HDR-METHOD with "GET"
#    HDR-POST        ( path-a path-u -- )    shortcut: HDR-METHOD with "POST"
#    HDR-PUT         ( path-a path-u -- )    shortcut: "PUT"
#    HDR-DELETE      ( path-a path-u -- )    shortcut: "DELETE"
#
#    HDR-HOST        ( host-a host-u -- )
#        Append "Host: hostname\r\n".
#
#    HDR-ADD         ( name-a name-u val-a val-u -- )
#        Append arbitrary "Name: Value\r\n" header.
#
#    HDR-AUTH-BEARER ( token-a token-u -- )
#        Append "Authorization: Bearer token\r\n".
#
#    HDR-CONTENT-TYPE ( ct-a ct-u -- )
#        Append "Content-Type: type\r\n".
#
#    HDR-CONTENT-JSON ( -- )
#        Shortcut: Content-Type: application/json
#
#    HDR-CONTENT-FORM ( -- )
#        Shortcut: Content-Type: application/x-www-form-urlencoded
#
#    HDR-CONTENT-LENGTH ( n -- )
#        Append "Content-Length: n\r\n".
#        Converts integer to decimal string.
#
#    HDR-CONNECTION-CLOSE ( -- )
#        Append "Connection: close\r\n".
#
#    HDR-ACCEPT      ( type-a type-u -- )
#        Append "Accept: type\r\n".
#
#    HDR-USER-AGENT  ( ua-a ua-u -- )
#        Append "User-Agent: string\r\n".
#
#    HDR-END         ( -- )
#        Append blank line "\r\n" (end of headers).
#
#    HDR-BODY        ( body-a body-u -- )
#        Append body data after headers.
#        Caller must have set Content-Length first.
#
#    HDR-RESULT      ( -- addr len )
#        Return the complete request (headers + body).
#
#
#  ── Parsing ──
#
#    HDR-FIND-HEND   ( addr len -- offset | 0 )
#        Find \r\n\r\n or \n\n header/body boundary.
#        Returns byte offset of body start, or 0 if not found.
#        Generalises tools.f _HTTP-FIND-HEND.
#
#    HDR-PARSE-STATUS ( addr len -- status-code )
#        Extract 3-digit HTTP status code from response line.
#        "HTTP/1.1 200 OK" → 200.
#
#    HDR-PARSE-CLEN  ( hdr-addr hdr-len -- content-length | -1 )
#        Find Content-Length header, return numeric value.
#        Case-insensitive.  Returns -1 if not found.
#
#    HDR-FIND        ( hdr-addr hdr-len name-a name-u -- val-a val-u flag )
#        Find any response header by name (case-insensitive).
#        Returns value string (trimmed) and flag.
#
#    HDR-CHUNKED?    ( hdr-addr hdr-len -- flag )
#        Is "Transfer-Encoding: chunked" present?
#
#    HDR-LOCATION    ( hdr-addr hdr-len -- url-a url-u flag )
#        Extract Location header for redirects.
#
#    HDR-SET-COOKIE  ( hdr-addr hdr-len -- val-a val-u flag )
#        Extract Set-Cookie header value (first one).
#
#
# =====================================================================
#  §4 — http.f — HTTP/1.1 Client
# =====================================================================
#
#  PROVIDED akashic-http
#  REQUIRE url.f
#  REQUIRE headers.f
#
#  Full HTTP/1.1 client with TLS support.  Replaces the HTTP/HTTPS
#  code in both tools.f and bsky.f with a single reusable implementation.
#
#  ── Layer 0 — Connection Management ──
#
#    HTTP-CONNECT    ( host-a host-u port tls? -- ctx | 0 )
#        Open TCP or TLS connection.  Sets SNI for TLS.
#        tls? flag: 0 = plain TCP, -1 = TLS.
#        Returns connection context (TCB or TLS handle).
#
#    HTTP-DISCONNECT ( ctx tls? -- )
#        Close connection.  Dispatches to TCP-CLOSE or TLS-CLOSE.
#
#    HTTP-SEND       ( ctx tls? buf len -- )
#        Send data.  Dispatches to TCP-SEND or TLS-SEND.
#
#    HTTP-RECV       ( ctx tls? buf max -- n )
#        Receive data.  Dispatches to TCP-RECV or TLS-RECV.
#        Returns bytes received (0 = nothing, -1 = error).
#
#
#  ── Layer 1 — Receive Loop ──
#
#    HTTP-RECV-BUF   VARIABLE   — address of receive buffer
#    HTTP-RECV-LEN   VARIABLE   — bytes received so far
#    HTTP-RECV-MAX   VARIABLE   — capacity of receive buffer
#
#    HTTP-ALLOC-RECV ( size -- )
#        Allocate receive buffer via ALLOCATE (or use HBW).
#        Sets HTTP-RECV-BUF/MAX.  Caller must HTTP-FREE-RECV later.
#
#    HTTP-FREE-RECV  ( -- )
#        Free receive buffer.
#
#    HTTP-USE-STATIC ( addr max -- )
#        Use a caller-provided static buffer instead of allocating.
#        For small responses (fits in SCROLL-BUF, etc.)
#
#    HTTP-RECV-LOOP  ( ctx tls? -- )
#        Receive full response into HTTP-RECV-BUF.
#        Polls with TCP-POLL + NET-IDLE.  Stops after 10 consecutive
#        empty reads or buffer full.  Same pattern as tools.f/bsky.f.
#
#
#  ── Layer 2 — Response Processing ──
#
#    HTTP-PARSE      ( -- ior )
#        Parse received response in HTTP-RECV-BUF.
#        Sets: HTTP-STATUS, HTTP-BODY-ADDR, HTTP-BODY-LEN.
#        Handles Content-Length and chunked transfer encoding.
#        Returns 0 on success.
#
#    HTTP-STATUS     VARIABLE   — last HTTP status code (200, 404, etc.)
#    HTTP-BODY-ADDR  VARIABLE   — pointer to body in recv buffer
#    HTTP-BODY-LEN   VARIABLE   — body length in bytes
#
#    HTTP-DECHUNK    ( addr len -- addr' len' )
#        Dechunk HTTP chunked transfer encoding in place.
#        Compacts data, returns new address and length.
#        Generalises bsky.f's _BSK-DECHUNK.
#
#    HTTP-HEADER     ( name-a name-u -- val-a val-u flag )
#        Find a response header by name in the last response.
#        Convenience wrapper around HDR-FIND.
#
#
#  ── Layer 3 — Request Execution ──
#
#    HTTP-REQUEST    ( method-a method-u url-a url-u -- ior )
#        Full request cycle:
#        1. URL-PARSE the URL
#        2. DNS-RESOLVE the host (cache IP)
#        3. HTTP-CONNECT (TLS if HTTPS)
#        4. Build + send request via HDR-*
#        5. HTTP-RECV-LOOP
#        6. HTTP-PARSE
#        7. HTTP-DISCONNECT
#        Returns 0 on success.  Body in HTTP-BODY-ADDR/LEN.
#
#    HTTP-GET        ( url-a url-u -- body-a body-u | 0 0 )
#        Convenience: GET request, return body.
#
#    HTTP-POST       ( url-a url-u body-a body-u ct-a ct-u -- resp-a resp-u | 0 0 )
#        Convenience: POST with Content-Type and body.
#
#    HTTP-POST-JSON  ( url-a url-u json-a json-u -- resp-a resp-u | 0 0 )
#        Convenience: POST with Content-Type: application/json.
#
#
#  ── Layer 4 — Session / Custom Headers ──
#
#    HTTP-SESSION state variables for persistent headers across requests:
#
#    HTTP-SET-BEARER ( token-a token-u -- )
#        Store Bearer token for Authorization header.
#        Automatically added to all subsequent requests.
#
#    HTTP-CLEAR-BEARER ( -- )
#        Clear stored Bearer token.
#
#    HTTP-SET-HOST   ( host-a host-u -- )
#        Override Host header (for virtual hosting / API endpoints).
#
#    HTTP-SET-UA     ( ua-a ua-u -- )
#        Set User-Agent for all requests.
#
#    HTTP-ADD-HEADER ( name-a name-u val-a val-u -- )
#        Add a custom header to be sent with every request.
#        Up to 8 custom headers stored.
#
#    HTTP-CLEAR-HEADERS ( -- )
#        Clear all custom session headers.
#
#
#  ── Layer 5 — DNS Cache ──
#
#    HTTP-DNS-CACHE  — 8-slot (host,ip) cache
#    HTTP-DNS-LOOKUP ( host-a host-u -- ip | 0 )
#        Check cache first, then DNS-RESOLVE, cache result.
#    HTTP-DNS-FLUSH  ( -- )
#        Clear DNS cache.
#
#
#  ── Layer 6 — Redirect Following ──
#
#    HTTP-MAX-REDIRECTS  VARIABLE  (default: 5)
#    HTTP-FOLLOW?        VARIABLE  (default: -1 = follow)
#
#    HTTP-GET redirects automatically:
#    On 301/302/307/308 → parse Location header → re-issue request
#    (up to HTTP-MAX-REDIRECTS hops)
#
#
#  Error handling:
#    HTTP-ERR            VARIABLE
#    HTTP-E-DNS          = 1    DNS resolution failed
#    HTTP-E-CONNECT      = 2    TCP/TLS connect failed
#    HTTP-E-SEND         = 3    send failed
#    HTTP-E-TIMEOUT      = 4    recv timeout (no data)
#    HTTP-E-PARSE        = 5    response parse error
#    HTTP-E-OVERFLOW     = 6    response exceeds buffer
#    HTTP-E-TLS          = 7    TLS decrypt error
#    HTTP-E-REDIRECT     = 8    too many redirects
#
#
# =====================================================================
#  §5 — websocket.f — WebSocket Client (RFC 6455)
# =====================================================================
#
#  PROVIDED akashic-websocket
#  REQUIRE http.f
#
#  WebSocket protocol for real-time bidirectional communication.
#  Uses HTTP for the upgrade handshake, then switches to framed mode.
#
#  ── Layer 0 — Handshake ──
#
#    WS-CONNECT      ( url-a url-u -- ctx | 0 )
#        Full WebSocket connection:
#        1. URL-PARSE (expect ws:// or wss://)
#        2. TCP or TLS connect
#        3. Send HTTP Upgrade request with Sec-WebSocket-Key
#        4. Validate 101 Switching Protocols response
#        Returns context handle.
#
#    WS-DISCONNECT   ( ctx -- )
#        Send close frame, tear down connection.
#
#    _WS-MAKE-KEY    ( -- addr len )
#        Generate 16-byte random nonce, Base64-encode to 24 chars.
#
#    _WS-VALIDATE    ( resp-a resp-u key-a key-u -- flag )
#        Validate Sec-WebSocket-Accept against our key.
#        accept = Base64(SHA-1(key + "258EAFA5-E914-..."))
#        Note: SHA-1 is only used here (not for crypto).
#
#
#  ── Layer 1 — Frame Encoding/Decoding ──
#
#    WS-FRAME-SEND   ( ctx opcode payload-a payload-u -- )
#        Build and send one WebSocket frame.  Client frames are
#        always masked (RFC 6455 §5.3).
#
#    WS-FRAME-RECV   ( ctx buf max -- opcode payload-a payload-u | -1 )
#        Receive and decode one frame.  Handles fragmentation
#        (continuation frames).  Returns -1 on error/close.
#
#    Opcodes:
#      WS-OP-TEXT     = 1     text frame
#      WS-OP-BINARY   = 2     binary frame
#      WS-OP-CLOSE    = 8     close control frame
#      WS-OP-PING     = 9     ping control frame
#      WS-OP-PONG     = 10    pong control frame
#
#    _WS-MASK        ( data len mask-key -- )
#        XOR mask data in place (4-byte rotating key).
#
#    _WS-ENCODE-LEN  ( len buf -- bytes-written )
#        Encode payload length into frame header (7, 7+16, or 7+64 bit).
#
#
#  ── Layer 2 — High-Level API ──
#
#    WS-SEND-TEXT    ( ctx text-a text-u -- )
#        Send text message (may fragment if > 4 KB).
#
#    WS-SEND-BINARY  ( ctx data-a data-u -- )
#        Send binary message.
#
#    WS-RECV         ( ctx buf max -- opcode len )
#        Receive next complete message (reassembles fragments).
#        opcode: 1 = text, 2 = binary.  Returns -1 on close.
#
#    WS-PING         ( ctx -- )
#        Send ping, expect pong.
#
#    WS-PONG         ( ctx payload-a payload-u -- )
#        Send pong (auto-sent on ping receipt by default).
#
#    WS-AUTO-PONG    VARIABLE  (default: -1 = auto-reply to pings)
#
#
# =====================================================================
#  §6 — ftp.f — FTP/FTPS Client
# =====================================================================
#
#  PROVIDED akashic-ftp
#
#  FTP client for file retrieval and upload.  Refactored from tools.f.
#  Supports implicit FTPS (TLS on connect) and explicit FTPS (AUTH TLS).
#
#    FTP-CONNECT     ( host-a host-u port ftps? -- ior )
#        Connect to FTP server.  Reads banner.
#        ftps? = -1: wrap control channel in TLS immediately.
#
#    FTP-LOGIN       ( user-a user-u pass-a pass-u -- ior )
#        Send USER + PASS.  Returns 0 on success.
#
#    FTP-LOGIN-ANON  ( -- ior )
#        Login as anonymous/guest@.
#
#    FTP-PASV        ( -- ip port ior )
#        Request passive mode, parse response for data channel.
#
#    FTP-RETR        ( path-a path-u buf max -- len ior )
#        Retrieve file into buffer.  Opens PASV data channel,
#        sends RETR, reads data, closes data channel.
#
#    FTP-STOR        ( path-a path-u data-a data-u -- ior )
#        Upload file.  Opens PASV data channel, sends STOR,
#        writes data, closes.
#
#    FTP-LIST        ( path-a path-u buf max -- len ior )
#        Directory listing via LIST.
#
#    FTP-CWD         ( path-a path-u -- ior )
#        Change remote directory.
#
#    FTP-PWD         ( buf max -- len )
#        Print working directory.
#
#    FTP-QUIT        ( -- )
#        Send QUIT, close connections.
#
#    FTP-AUTH-TLS    ( -- ior )
#        Upgrade control channel to TLS (explicit FTPS).
#
#  Error codes:
#    FTP-E-CONNECT   = 1
#    FTP-E-LOGIN     = 2
#    FTP-E-PASV      = 3
#    FTP-E-TRANSFER  = 4
#    FTP-E-COMMAND   = 5
#

# =====================================================================
#  §9 — ntp.f — NTP Time Sync (SNTP)
# =====================================================================
#
#  PROVIDED akashic-ntp
#
#  Simple Network Time Protocol (RFC 5905, SNTP client mode).
#  Queries an NTP server, adjusts the KDOS RTC.
#
#    NTP-QUERY       ( server-ip -- epoch-secs ior )
#        Send NTP request, receive response, extract timestamp.
#        Returns Unix epoch seconds and 0 on success.
#
#    NTP-SYNC        ( server-ip -- ior )
#        Query NTP and set the KDOS RTC.
#
#    NTP-SYNC-DNS    ( "hostname" -- ior )
#        Resolve hostname, then NTP-SYNC.
#        Default: "pool.ntp.org"
#
#    NTP-AUTO        ( -- )
#        Sync from pool.ntp.org using DNS.  (One-shot.)
#
#  Internal:
#    _NTP-BUILD-PKT  ( -- )    Build 48-byte NTP request packet
#    _NTP-PARSE-RESP ( buf -- epoch-secs ior )  Parse NTP response
#    _NTP-EPOCH-OFFSET CONSTANT  2208988800
#        NTP epoch (1900-01-01) to Unix epoch (1970-01-01) offset.
#

# =====================================================================
#  §10 — irc.f — IRC Client (RFC 2812)
# =====================================================================
#
#  PROVIDED akashic-irc
#
#  IRC (Internet Relay Chat) client.  Text-based protocol that fits
#  beautifully on a Forth system with 80-column display.
#  Supports plain TCP and TLS (IRCS, port 6697).
#
#  ── Layer 0 — Connection ──
#
#    IRC-CONNECT     ( host-a host-u port tls? -- ior )
#        Connect to IRC server.
#
#    IRC-DISCONNECT  ( -- )
#        Send QUIT and close.
#
#    IRC-REGISTER    ( nick-a nick-u user-a user-u real-a real-u -- ior )
#        Send NICK + USER registration.
#
#
#  ── Layer 1 — Raw Protocol ──
#
#    IRC-SEND-RAW    ( line-a line-u -- )
#        Send raw IRC line (adds \r\n automatically).
#
#    IRC-RECV-LINE   ( buf max -- len | 0 )
#        Receive one line.  Returns 0 if no data.
#
#    IRC-PARSE-MSG   ( line-a line-u -- prefix-a prefix-u
#                      cmd-a cmd-u  params-a params-u )
#        Parse IRC message into prefix, command, and params.
#
#
#  ── Layer 2 — Commands ──
#
#    IRC-JOIN        ( channel-a channel-u -- )      JOIN #channel
#    IRC-PART        ( channel-a channel-u -- )      PART #channel
#    IRC-PRIVMSG     ( target-a target-u msg-a msg-u -- )
#    IRC-NOTICE      ( target-a target-u msg-a msg-u -- )
#    IRC-NICK        ( nick-a nick-u -- )
#    IRC-QUIT        ( msg-a msg-u -- )
#    IRC-PING-REPLY  ( server-a server-u -- )        PONG response
#    IRC-WHO         ( target-a target-u -- )
#    IRC-WHOIS       ( nick-a nick-u -- )
#    IRC-MODE        ( target-a target-u mode-a mode-u -- )
#    IRC-TOPIC       ( channel-a channel-u -- )      query topic
#    IRC-TOPIC-SET   ( channel-a channel-u topic-a topic-u -- )
#
#
#  ── Layer 3 — Event Loop ──
#
#    IRC-POLL        ( -- )
#        Check for incoming data, auto-handle PING/PONG.
#        Calls user callback on messages.
#
#    IRC-ON-MSG      VARIABLE   — xt of callback
#        Callback signature: ( prefix-a prefix-u cmd-a cmd-u
#                              params-a params-u -- )
#        Set with:  ' my-handler IRC-ON-MSG !
#
#    IRC-RUN         ( -- )
#        Main event loop: IRC-POLL + KEY? handling.
#        Press 'q' to quit.

# =====================================================================
#  §11 — rabbit.f — Rabbit Protocol Client
# =====================================================================
#
#  PROVIDED akashic-rabbit
#
#  Rabbit protocol (rabbit://), a text-based, peer-to-peer,
#  asynchronous protocol for federated networks of burrows.
#  Gopher-like simplicity (human-readable UTF-8 frames, tab-delimited
#  menus) layered with TLS 1.3 transport and Ed25519 identity.
#  See https://github.com/dgoldman0/Rabbit/ for the full spec.
#
#  Wire format: all frames are UTF-8 with CRLF line endings:
#    VERB [args]\r\n
#    Header: Value\r\n
#    End:\r\n
#    [body]
#
#  ── Layer 0 — Frame Serialization ──
#
#    RBT-FRAME-RESET   ( -- )
#        Clear the frame builder for a new frame.
#
#    RBT-FRAME-SET-OUT ( addr max -- )
#        Set output buffer for frame building.
#        Default: internal 2 KB buffer.
#
#    RBT-FRAME-VERB    ( verb-a verb-u -- )
#        Set the verb line: "VERB [args]\r\n".
#
#    RBT-FRAME-HEADER  ( name-a name-u val-a val-u -- )
#        Append "Name: Value\r\n".
#
#    RBT-FRAME-END     ( -- )
#        Append "End:\r\n".
#
#    RBT-FRAME-BODY    ( body-a body-u -- )
#        Append body after End: marker.  Automatically sets
#        Length header before End:.
#
#    RBT-FRAME-RESULT  ( -- addr len )
#        Return the serialised frame.
#
#    RBT-FRAME-PARSE   ( data-a data-u -- ior )
#        Parse a received frame.  Sets:
#        RBT-VERB, RBT-ARGS, RBT-BODY-ADDR, RBT-BODY-LEN.
#        Headers accessible via RBT-FRAME-FIND.
#
#    RBT-VERB          CREATE 32 ALLOT   — parsed verb string
#    RBT-VERB-LEN      VARIABLE
#    RBT-ARGS          CREATE 256 ALLOT  — parsed args string
#    RBT-ARGS-LEN      VARIABLE
#    RBT-BODY-ADDR     VARIABLE
#    RBT-BODY-LEN      VARIABLE
#
#    RBT-FRAME-FIND    ( name-a name-u -- val-a val-u flag )
#        Find a header in the last parsed frame by name.
#
#    RBT-STATUS        ( -- n )
#        Extract numeric status code from verb ("200" → 200).
#
#
#  ── Layer 1 — Connection & Handshake ──
#
#    RBT-CONNECT       ( host-a host-u port -- ior )
#        TLS connect to a burrow.  Sets SNI.
#        Default port: 7443.
#
#    RBT-DISCONNECT    ( -- )
#        TLS close.
#
#    RBT-SEND-FRAME    ( addr len -- )
#        Send serialised frame via TLS.
#
#    RBT-RECV-FRAME    ( buf max -- len ior )
#        Receive one frame from TLS.  Reads lines until End:\r\n,
#        then reads Length bytes of body if present.
#
#    RBT-HELLO         ( -- ior )
#        Send HELLO RABBIT/1.0 with Burrow-ID and Caps headers.
#        For anonymous connections, sends minimal HELLO.
#        Receives 200 HELLO (or 300 CHALLENGE → AUTH → 200).
#        Sets RBT-SERVER-ID.
#
#    RBT-SERVER-ID     CREATE 64 ALLOT   — remote burrow ID
#    RBT-SERVER-ID-LEN VARIABLE
#    RBT-SESSION-TOKEN CREATE 64 ALLOT   — session token
#    RBT-SESSION-LEN   VARIABLE
#
#    RBT-ANONYMOUS?    VARIABLE  (default: -1 = anonymous mode)
#        When false, full Ed25519 auth is attempted.
#
#
#  ── Layer 2 — Protocol Verbs ──
#
#    RBT-LIST          ( sel-a sel-u buf max -- len ior )
#        Send LIST <selector>, receive 200 MENU response.
#        Body is a rabbitmap (tab-delimited menu).
#
#    RBT-FETCH         ( sel-a sel-u buf max -- len ior )
#        Send FETCH <selector>, receive 200 CONTENT response.
#        Returns body text.
#
#    RBT-SUBSCRIBE     ( topic-a topic-u -- ior )
#        Send SUBSCRIBE <topic>.  Receive 201 SUBSCRIBED.
#        Subsequent RBT-RECV-FRAME calls yield EVENT frames.
#
#    RBT-PUBLISH       ( topic-a topic-u msg-a msg-u -- ior )
#        Send PUBLISH <topic> with body.  Receive 204 DONE.
#
#    RBT-PING          ( -- ior )
#        Send PING, receive 200 PONG.
#
#
#  ── Layer 3 — Menu Parsing ──
#
#    Rabbit menus (rabbitmaps) use Gopher-style tab-delimited lines:
#      <type><label>\t<selector>\t<burrow>\t<hint>\r\n
#    Menu terminates with a line containing only "."
#
#    RBT-MENU-NEXT     ( a u -- a' u' type label-a label-u
#                        sel-a sel-u burrow-a burrow-u flag )
#        Parse one menu line.  Returns all fields + flag.
#        Type: '0'=text, '1'=menu, 'q'=event stream, 'i'=info.
#        flag = -1 found, 0 end (".").
#
#    RBT-MENU-TYPE     ( type-char -- )
#        Print human-readable type label ("[text]", "[menu]", etc.)
#
#
#  ── Layer 4 — High-Level Browse ──
#
#    RBT-BROWSE        ( host-a host-u port -- )
#        Interactive burrow browser:
#        1. RBT-CONNECT + RBT-HELLO
#        2. RBT-LIST / to get root menu
#        3. Display numbered menu items
#        4. User types number → navigate, 'b' → back, 'q' → quit
#        5. Text items displayed with FETCH
#        6. Sub-menus navigated with LIST
#        Maintains a back-stack for navigation.
#
#    RBT-URL-FETCH     ( url-a url-u buf max -- len ior )
#        Parse rabbit:// URL, connect, hello, fetch, disconnect.
#        One-shot convenience for SCROLL integration.
#
#
#  Lane/Txn management (simplified for single-lane client):
#    RBT-LANE          VARIABLE  (default: 1)
#    RBT-TXN-COUNTER   VARIABLE  (starts at 0)
#    RBT-NEXT-TXN      ( -- addr len )  "T-<n>" string, increments counter.
#
#  Error handling:
#    RBT-ERR           VARIABLE
#    RBT-E-CONNECT     = 1    TLS connect failed
#    RBT-E-HELLO       = 2    handshake failed
#    RBT-E-SEND        = 3    send failed
#    RBT-E-RECV        = 4    recv failed / timeout
#    RBT-E-STATUS      = 5    unexpected status code
#    RBT-E-PARSE       = 6    frame parse error
#    RBT-E-AUTH        = 7    authentication failed
#
#  Response status constants:
#    RBT-S-OK          = 200
#    RBT-S-SUBSCRIBED  = 201
#    RBT-S-DONE        = 204
#    RBT-S-CHALLENGE   = 300
#    RBT-S-MOVED       = 301
#    RBT-S-BAD-REQ     = 400
#    RBT-S-FORBIDDEN   = 403
#    RBT-S-MISSING     = 404
#    RBT-S-TIMEOUT     = 408
#    RBT-S-AUTH-REQ    = 440
#    RBT-S-BUSY        = 503
#
#

# =====================================================================
#  Key Design Decisions
# =====================================================================
#
#  1. Same (addr len) cursor model as json.f, markup/core.f, css.f
#     for all text parsing (URL parsing, header parsing, etc.)
#
#  2. Each file is independently REQUIRE-able with PROVIDED guards.
#     No hidden implicit loads except documented dependencies.
#
#  3. Error handling per-library: URL-ERR, HTTP-ERR, FTP-ERR, etc.
#     Same VARIABLE + FAIL + OK? pattern as all other akashic libs.
#
#  4. http.f is the only file with cross-library REQUIREs
#     (url.f + headers.f).  All protocol clients (ftp, gopher,
#     tftp, ntp, irc, rabbit) are standalone — they talk directly
#     to KDOS TCP/UDP/TLS, not through http.f.
#
#  5. Receive buffers:
#     - Static mode: caller provides buffer (HTTP-USE-STATIC)
#     - Dynamic mode: library ALLOCATEs (HTTP-ALLOC-RECV)
#     - bsky.f pattern of HBW-backed large recv buffer is
#       available via HTTP-ALLOC-RECV with large size.
#
#  6. No floating point.  NTP timestamps and HTTP dates use
#     integer epoch seconds.
#
#  7. HTTP session state (bearer token, custom headers) uses
#     simple VARIABLEs and static buffers — no dynamic alloc.
#     Max 8 custom headers × 256 bytes each.
#
#  8. WebSocket masking key comes from TRNG (hardware RNG).
#
#  9. DNS cache is 8 slots, LRU eviction.  Enough for typical
#     usage where an app talks to 1-3 servers.
#
# 10. The SCROLL system in tools.f becomes a thin dispatcher:
#       URL-PARSE → protocol switch → fetch → SCROLL-BUF.
#     All protocol-specific code moves to net/*.
#     SCROLL public API unchanged — no breaking changes.
#
# 11. Protocol clients that use TCP directly (FTP, Gopher, IRC)
#     include their own recv-line helpers because IRC needs
#     line-at-a-time, FTP needs response-code parsing, etc.
#     No shared "line protocol" abstraction — it would be
#     over-engineered for the minor duplication.
#
# 12. Prefix choice rationale:
#     - HTTP- (not HT-) because HTTP is universally recognizable
#     - URL- (not URI-) because URLs are what users type
#     - IRC- (not IR-) follows the protocol's own name
#     - GOPH- (not GO-) avoids conflict with Forth GO
#     - RBT- for Rabbit (concise, unambiguous)
#     - SMTP- (not SM-) follows the protocol's own name
#     - WS- for WebSocket (concise, unambiguous in context)
#

# =====================================================================
#  Notes on Specific Protocol Considerations
# =====================================================================
#
#  HTTP:
#  - Connection: close by default (no keep-alive / pipelining).
#    One request per connection.  Simple, predictable, works with
#    the 4-TCB limit.
#  - Chunked transfer decoding is critical — many real servers
#    use it (bsky.social does).  The bsky.f implementation is
#    battle-tested.
#  - Redirect following is important for HTTPS — many sites
#    redirect http→https or www→bare domain.
#
#  WebSocket:
#  - Client MUST mask frames (RFC 6455 §5.3).  Uses TRNG for
#    the 4-byte mask key.
#  - SHA-1 is needed only for the handshake validation hash.
#    KDOS doesn't have SHA-1 (it has SHA-256 and SHA-3).
#    Options: (a) implement minimal SHA-1 (~80 lines), (b) skip
#    validation (not recommended), (c) use SHA-256 if server
#    accepts it (non-standard).
#    Decision: implement minimal SHA-1 inside websocket.f.
#
#  NTP:
#  - SNTP is sufficient — we don't need full NTP with
#    drift correction.  Just set the RTC once.
#  - 48-byte request, 48-byte response.  UDP.  Very simple.
#  - NTP epoch starts 1900; Unix epoch starts 1970.
#    Offset: 2208988800 seconds.
#
#  IRC:
#  - Line-oriented protocol: "COMMAND params :trailing\r\n"
#  - Must auto-respond to PING with PONG (server keepalive).
#  - Perfect for a Forth system: text-mode, 80 columns,
#    interactive, line-at-a-time.
#  - TLS support (IRCS on port 6697) is important — most
#    modern IRC networks require it.
#
#  Rabbit:
#  - Always TLS (port 7443).
#  - Text-based frames: VERB + headers + End: + body.  UTF-8, CRLF.
#  - Gopher-like tab-delimited menus (rabbitmaps) with type codes.
#  - Ed25519 identity + TOFU trust.  Anonymous connections supported.
#  - Pub/sub with event streams is native (SUBSCRIBE/PUBLISH/EVENT).
#  - Perfect for Forth: text-only wire format, human-readable,
#    line-at-a-time parsing, 80-column menus.
#  - Client is a full peer with an ephemeral identity per session.
#  - Lanes provide async multiplexing, but our client uses one lane.
#
#  SMTP:
#  - STARTTLS is needed for almost all real mail servers.
#  - AUTH LOGIN sends Base64-encoded username and password
#    as separate lines.  Requires base64.f.
#  - Dot-stuffing: lines starting with "." get an extra ".".
#  - The emulator's TAP bridge with masquerade + IP forwarding
#    allows real SMTP connections to external servers.
#
#  FTP:
#  - PASV response parsing is tricky: "(h1,h2,h3,h4,p1,p2)"
#    where port = p1*256 + p2.
#  - Active mode (PORT) is not supported — PASV only.
#    Active mode requires incoming connections which don't work
#    well behind NAT.
#  - Directory listings (LIST) return free-form text, not
#    machine-parseable.  We just display it.
