# Refactor Landing L0 architecture baseline

Landing L0 records the post-Gate-4 ownership, dependency, capacity, mutable-state,
and scale baseline before production refactoring begins. It changes no production
Forth behavior.

The post-L0 scope amendment made during plan review keeps the measured graph,
state, unresolved-import facts, counts, and production sources intact while
correcting future ownership decisions, workload scope, and landing labels. The
placement digest changes only because the shared-document lens target moves from
a second Desk-specific session to the neutral interop resource-session seam. The
amendment changes no production Forth behavior.

The reviewed machine policy is
`local_testing/refactor_architecture.json`. The live graph and inventory are
generated by `local_testing/refactor_inventory.py`; the JSON report is the
machine-readable graph and includes every production module, every resolved and
unresolved `REQUIRE` occurrence, ownership/target placement, exact lexical
mutable symbols, declared capacities, and reviewed scale profiles.

## Settled ownership model

There are three active ownership classes:

1. Independent Akashic libraries contain neutral mechanisms, caller-owned state,
   injected policy, direct tests, and real independent consumers.
2. Shared Desk/TUI facilities live under `akashic/tui/` and import no concrete
   applet.
3. Applet-owned models, repositories, services, adapters, actions, and views live
   under `akashic/tui/applets/<name>/`.

Renderer-free testability does not create a standalone product domain. The live
ledger consequently classifies:

- all five top-level Library modules as misplaced Library-applet code;
- the concrete Daybook shared-document owner as misplaced Daybook-applet code;
- the complete top-level Agent tree as misplaced Agent-applet code: its
  renderer-free service/provider/storage internals remain useful subpackages,
  but no current file has an independent non-Agent consumer;
- `agent/access-profile.f` as a required split: its Agent profile contract moves
  with the applet, while the concrete `desk.*` access presets move into Desk
  product policy;
- the shared-document lens as transitional resource-session machinery whose
  neutral successor remains under interop;
- Streams source/observation/store/refresh modules as correctly applet-owned;
- Agent authentication/settings widgets as misplaced Agent-applet widgets.

Game is the explicit exception to final ownership settlement. `game/` retains
its current independent dependency class and `tui/game/` retains its current
Desk/TUI dependency class, but both carry
`ownership_decision: deferred-current-placement`. Neither tree is active
refactor scope or a proving consumer, and the policy makes no claim about their
future package architecture.

The target plan does not retain compatibility facades at the old product paths.

## Capacity baseline

The machine ledger verifies literal production symbols rather than copying
unlinked numbers into prose. Each entry distinguishes its scope and scale axis:

- aggregate corpus/relationship/catalog limits must be replaced by indexed,
  storage-backed cardinality;
- whole-corpus, whole-thread, and whole-document buffers require paging,
  chunking, or streaming;
- working-set, viewport, and service-pool bounds may remain when caller-selected,
  observable, and paired with explicit backpressure;
- field, protocol, and safety bounds are not automatically scalability defects.

The initial ledger covers Library, Streams, Agent, Daybook, Pad, Grid,
FExplorer, Desk, runtime registries, request bus, UIDL, app-shell, and tree
limits. The JSON report fails if any named literal silently changes.

A separate source-anchored complexity ledger records the current structure,
operation, complexity, scale failure, replacement requirement, and landing for
13 major hot paths. Every claim names live Forth words that must still exist;
the ratchet fails if prose drifts away from the source being described. Library
entries target L12 and Streams targets L13. Agent, Daybook, Pad, FExplorer,
Grid, Desk-catalog, runtime-registry, request-bus, and UIDL scale directions
remain recorded but explicitly deferred.

## Scale profiles and units

The active qualification targets are applet-specific:

| Profile | Shape |
| --- | --- |
| Library workstation | 100,000 documents, at least 1,000,000 revisions or relationships, and content beyond 64 KiB |
| Library large host/model | 1,000,000 documents, 10,000,000 revisions, and 10,000,000 relationship edges |
| Streams workstation | 10,000 sources, 1,000,000 observations, and 2,000,000 retained attempts |
| Streams large host/model | 100,000 sources, 10,000,000 observations, and 20,000,000 retained attempts |
| Emulator | Reduced applet-specific datasets at structural transitions preserving index-height, page-count, amplification, working-set, and fault invariants |

The profile also fixes query and complexity shapes: point lookup, ordered and
compound ranges, relationship neighborhoods, text candidate verification, and
deep keyset pagination; variable per-record metadata; sparse, skewed, and
high-degree relationships; and variable/chunked content. Streams additionally
qualifies skewed source histories, retained attempts, bounded refresh batches,
and deep time/source/thread continuation.

Active reentrancy qualification is scoped to four interleaved converted stores
with no hidden process-global current store. Ecosystem-wide same-type applet,
request-bus, long-job, and 100,000-route targets are not active requirements;
their current capacities remain inventoried as facts for possible later work.

Three units must remain distinct:

- current MP64FS/device sectors are 512 bytes;
- current Library query pages contain 32 result summaries;
- future persistence pages do not yet have a selected byte size.

The existing profilers report exact `PERF-CYCLES`, modeled stalls, external
memory traffic, and Library-specific validation/index/read/scan counters.
`PERF-EXTMEM` is not allocation telemetry and currently does not provide evidence
about allocation events or transient peak memory. Physical device I/O,
persistence-page I/O, allocation events, and peak live allocation remain named
instrumentation gaps; they are added with the replacement engine rather than
invented from sector counts.

Only Library currently has isolated cycle/operation telemetry. Streams, Agent,
the other data applets, and Desk/TUI are explicitly marked as measurement gaps;
L0 inventories that absence instead of presenting journey tests as counters.

Committed reference points include:

| Current operation | Shape | Guest cycles |
| --- | --- | ---: |
| projection replace | 65,536-byte content | 495,847,848 |
| maintenance inspect | nine 65,536-byte payloads | 546,637,946 |
| maintenance repair | nine 65,536-byte payloads | 1,627,300,611 |

These characterize machinery scheduled for replacement; they are not preserved
as compatibility targets.

## Ratchet update rule

A changed digest is not regenerated mechanically. The landing must inspect the
JSON diff and explain which dependency, placement, unresolved import, module,
or mutable symbol changed. The reviewed expected digest is updated only when:

- the change moves ownership in the settled direction;
- a new neutral module has proving consumers and no upward dependency;
- a named debt is removed;
- or a deliberately introduced dependency/state item has an explicit owner and
  removal/scaling plan.

Unknown top-level source prefixes classify as `unclassified` and fail the
responsibility test. Existing debt is exact, so deleting it is encouraged while
reintroducing or widening it fails.
